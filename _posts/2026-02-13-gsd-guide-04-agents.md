## 에이전트 아키텍처

GSD는 **얇은 오케스트레이터가 전문 에이전트를 스폰**하는 패턴을 사용합니다:

```
┌─────────────────────────────────────────────────────────────┐
│                    오케스트레이터                            │
│         (얇은 레이어 - 조정만 담당)                          │
├─────────────────────────────────────────────────────────────┤
│                      ↓ 스폰                                  │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐        │
│  │ 리서처   │ │ 플래너   │ │ 실행자   │ │ 검증자   │        │
│  │ (병렬)   │ │ (병렬)   │ │ (병렬)   │ │ (병렬)   │        │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘        │
│                      ↓ 수집                                  │
│              결과 통합 및 라우팅                             │
└─────────────────────────────────────────────────────────────┘
```

오케스트레이터는 무거운 작업을 하지 않습니다. 에이전트를 스폰하고, 대기하고, 결과를 통합합니다.

---

## 에이전트 목록

GSD는 11개의 전문 에이전트를 제공합니다:

| 에이전트 | 역할 | 스폰 주체 |
|----------|------|----------|
| `gsd-planner` | 실행 가능한 단계 계획 생성 | plan-phase 오케스트레이터 |
| `gsd-executor` | 계획 실행, 원자적 커밋 | execute-phase 오케스트레이터 |
| `gsd-verifier` | 코드베이스 검증 | verify-work 오케스트레이터 |
| `gsd-debugger` | 체계적 디버깅 | debug 명령어 |
| `gsd-plan-checker` | 계획 품질 검증 | plan-phase 오케스트레이터 |
| `gsd-phase-researcher` | 단계별 리서치 | plan-phase 오케스트레이터 |
| `gsd-project-researcher` | 프로젝트 리서치 | new-project 오케스트레이터 |
| `gsd-research-synthesizer` | 리서치 결과 종합 | 리서치 오케스트레이터 |
| `gsd-roadmapper` | 로드맵 생성 | new-project 오케스트레이터 |
| `gsd-codebase-mapper` | 기존 코드 분석 | map-codebase 명령어 |
| `gsd-integration-checker` | 통합 검증 | execute-phase 오케스트레이터 |

## 플래너 (gsd-planner)

**역할:** 실행 가능한 단계 계획 생성

**핵심 책임:**
- CONTEXT.md의 사용자 결정 파싱 및 준수
- 단계를 병렬 최적화된 계획으로 분해 (각 2-3개 태스크)
- 의존성 그래프 구축 및 실행 웨이브 할당
- 목표-역방향 방법론으로 필수 기능 도출

### 계획 품질 곡선

| 컨텍스트 사용량 | 품질 | Claude 상태 |
|----------------|------|------------|
| 0-30% | 최고 | 철저하고 종합적 |
| 30-50% | 양호 | 자신감 있고 견고 |
| 50-70% | 저하됨 | 효율 모드 시작 |
| 70%+ | 나쁨 | 성급하고 최소화 |

**규칙:** 계획은 약 50% 컨텍스트 내에서 완료되어야 합니다. 더 많은 계획, 더 작은 범위, 일관된 품질.

### 태스크 분해 원칙

각 태스크는 4개 필수 필드를 가집니다:

```xml
<task type="auto">
  <files>정확한 파일 경로</files>
  <action>구체적인 구현 지침</action>
  <verify>완료 증명 방법</verify>
  <done>수락 기준</done>
</task>
```

## 실행자 (gsd-executor)

**역할:** 계획을 원자적으로 실행

**실행 패턴:**

| 패턴 | 설명 |
|------|------|
| **A: 완전 자율** | 체크포인트 없음, 모든 태스크 실행 후 SUMMARY 생성 |
| **B: 체크포인트 있음** | 체크포인트까지 실행 후 정지 |
| **C: 계속** | 이전 태스크 확인 후 지정된 태스크부터 재개 |

### 이탈 규칙 (Deviation Rules)

실행 중 계획에 없는 작업을 발견할 때 자동으로 적용:

| 규칙 | 트리거 | 동작 |
|------|--------|------|
| **Rule 1** | 코드가 의도대로 작동하지 않음 | 자동 수정 |
| **Rule 2** | 필수 기능 누락 (보안, 검증 등) | 자동 추가 |
| **Rule 3** | 차단 이슈 | 자동 해결 |
| **Rule 4** | 아키텍처 변경 필요 | **정지 후 사용자 결정** |

**규칙 우선순위:**
1. Rule 4 적용 → 정지 (아키텍처 결정)
2. Rule 1-3 적용 → 자동 수정
3. 확실하지 않음 → Rule 4 (질문)

## 검증자 (gsd-verifier)

**역할:** 코드베이스가 목표를 달성했는지 검증

**검증 항목:**
- 코드 존재 확인
- 테스트 통과 확인
- 기능적 요구사항 충족
- 비기능적 요구사항 충족

---

## 디버거 (gsd-debugger)

**역할:** 체계적인 디버깅

**특징:**
- 영구 상태로 디버깅 진행
- 근본 원인 분석
- 수정 계획 생성

---

## 단계 리서처 (gsd-phase-researcher)

**역할:** 각 단계의 구현 방법 조사

**조사 영역:**
- 스택 분석
- 기능 조사
- 아키텍처 분석
- 잠재적 문제점 식별

---

## 계획 체커 (gsd-plan-checker)

**역할:** 계획 품질 검증

**검증 항목:**
- 계획이 단계 목표를 달성하는지
- 태스크가 충분히 구체적인지
- 의존성이 올바르게 매핑되었는지

---

## 오케스트레이션의 이점

**결과:** 전체 단계를 실행할 수 있습니다 — 깊은 리서치, 여러 계획 생성 및 검증, 병렬 실행자로 수천 줄의 코드 작성, 자동화된 목표 검증 — 그리고 **메인 컨텍스트 윈도우는 30-40%에 유지**됩니다.

작업은 **새로운 서브에이전트 컨텍스트**에서 발생합니다. 세션은 빠르고 반응적으로 유지됩니다.
